## 0. What this milestone is — and is not

M9 is a **conceptual boundary-drawing** exercise. It produces a shared vocabulary and a classification of every top-level artifact in the repository into one of five quarantine categories, so operators and Ori can tell **active operational truth** apart from **runtime artifacts**, **historical archives**, **research outputs**, and **temporary snapshots** — *by looking, not by guessing*.

It deliberately stops at the boundary. It does not move, rename, delete, compress, or relocate anything, and it changes no reader/writer path.

> **M9 defines boundaries. M10 defines policies. A1 later defines architecture. Those are three different responsibilities.**

| Milestone | Question it answers | This sprint? | What it may do |
|-----------|---------------------|--------------|----------------|
| **M9** (this doc) | *"What category is each artifact, and where is the line between active and not?"* | Sprint 3 | Classify and label only. **No moves.** |
| **M10** | *"What do we do with each category — retire, archive, keep?"* | Sprint 3 (after M9) | Define retirement/quarantine **policy**; may then move legacy scanners to `archive/`, set single entry point. |
| **A1** | *"How should state and packaging be structured so categories are enforced by design?"* | Sprint 4 | Unified state layer, atomic writes, `data/` packaging, single-writer model. |

If a proposed action moves a file, renames a path, or changes what a process reads/writes, **it is not M9** — it belongs to M10 or A1.

## 1. Why boundaries first (the failure mode being prevented)

From [STABILIZATION_PLAN.md](./STABILIZATION_PLAN.md) (rank 4) and [KNOWN_ISSUES.md](./KNOWN_ISSUES.md) "Duplicate code trees":

- Full/partial copies of `live_executor.js`, `monitor.js`, `dashboard_server.js`, and `live_config.json` exist in `automation/`, `hardreset/`, `harness/`, `files/`, and `phase1_files/` **beside** the canonical root scripts.
- Legacy root scanners (`scanner.js`, `scanner_v3.js`, `scanner_trending.js`) and `*_backup.js` copies sit next to the one active scanner (`scanner_gmgn_trending.js`).
- **Impact:** fixes applied to the wrong tree; tests passing against the wrong copy; migration packaging the wrong code; operators editing a dead file and believing the bot changed.

[DECISIONS.md](./DECISIONS.md) already records the canonical-path decision: *"Active runtime path is project root … Folders `automation/`, `hardreset/`, `harness/`, `files/`, `phase1_files/` are archival duplicates, not second deployments,"* with future review trigger *"Archive or quarantine duplicate trees."* M9 is the disciplined first half of that trigger: **name the boundaries precisely before anyone acts on them.**

The Sprint 2 lesson applies directly: **visibility before enforcement.** Just as read-only panels delivered value without touching gate logic, M9 delivers a clean mental model without touching the filesystem.

## 2. Quarantine categories (definitions)

Five categories. Each artifact gets exactly one **primary** category (overlaps noted where real).

| Category | Definition | Git posture today | Safe to delete? |
|----------|-----------|-------------------|-----------------|
| **ACTIVE** | Operational truth: source code, config, docs, ops/safety scripts that a bot started from repo root actually uses. | Tracked | **No** |
| **RUNTIME LOCAL** | Environment-specific operational history written by live processes (ledgers, audit, telemetry, snapshots). Regenerated by running; not source. | **Gitignored** | Only with operator intent (loses history) |
| **ARCHIVE** | Historical or duplicate code trees and packaged exports kept for provenance. Not a second deployment. | Mostly gitignored (`*.zip`); folders tracked | Eventually (M10 policy) |
| **RESEARCH** | Analytical outputs and rejected-candidate datasets used for study, not for execution. | Gitignored | Yes, but valuable for analysis |
| **TEMPORARY** | Ad-hoc backups and point-in-time snapshots (`*_backup.*`, `*_before_*.json`, snapshot zips). | Gitignored | Yes (lowest value) |

These categories are **descriptive labels**, not directories. M9 does not create folders for them.

## 4. Quarantine boundary (the line)

The boundary M9 draws, stated plainly:

1. **ACTIVE is exactly what [ACTIVE_MANIFEST.md](../ACTIVE_MANIFEST.md) lists as canonical at repo root, plus `docs/`, `tools/`, CI, dependency manifests, and dashboard assets.** Nothing in `automation/`, `hardreset/`, `harness/`, `files/`, `phase1_files/` is ACTIVE.
2. **A file's category is determined by role, not location.** A legacy scanner sitting in root (`scanner_v3.js`) is ARCHIVE; an active ledger sitting in root (`paper_trades.json`) is RUNTIME LOCAL. Location today is not destiny — that is precisely the confusion M9 names so M10/A1 can later fix it.
3. **`.gitignore` already encodes most of the boundary** (it ignores `archive/`, `backups/`, `*.zip`, `*_backup.json`, `*_before_*.json`, and the runtime ledgers). M9 makes that implicit boundary **explicit and human-readable**; it does not change `.gitignore`.
4. **The boundary is advisory in M9.** Enforcement (read-only markers, moves, single entry point) is M10; structural enforcement (packaging, `data/`) is A1.

## 5. Migration principles (carry into M10 / A1)

1. **Repo wins.** The repository root + ACTIVE_MANIFEST is the single source of truth; archives never override it.
2. **Classify before you move.** No artifact is relocated until it has an agreed category (this doc) and a policy (M10).
3. **Provenance before deletion.** Every ARCHIVE/TEMPORARY artifact is reviewed for provenance before any retirement; nothing is deleted "to clean up."
4. **One change, one tree.** Edits land only in canonical root files; never patch a duplicate tree.
5. **Move readers and writers together, never alone.** Any future path change (A1 `data/`) updates every reader/writer atomically — not piecemeal.
6. **Gitignore hides, it does not delete.** Removing an artifact from `git status` is not retirement; physical handling is an explicit, logged action.
7. **Label estimates and dead data honestly.** `live_trades.json` (legacy) and pre-bot10 snapshots are named as superseded, not silently kept or silently purged.
8. **Every structural change earns a DECISIONS entry.** Consistent with the existing canonical-path decision and its review trigger.

- ❌ **Do not change any reader/writer path** in scanner, monitor, executor, dashboard, or wallet monitor.
- ❌ **Do not edit `.gitignore`, `live_config.json`, `PIPELINE_DRY_RUN`, or any executor/strategy logic.**
- ❌ **Do not create an `archive/` or `data/` directory** (that is M10 / A1).
- ❌ **Do not add read-only file attributes or markers** (that is M10 enforcement).
- ❌ **Do not introduce a database, index, or new dependency.**
- ✅ **Allowed:** writing/maintaining this classification doc and (optionally) referencing it from `ACTIVE_MANIFEST.md`'s existing "Sprint 2+ (M9)" note in a later doc-only edit.

If acting on M9 ever requires running a shell command that moves/renames/deletes a file, **stop — that is out of scope.**

## 11. Relationship to A1 (unified state architecture)

- The RUNTIME LOCAL category is the input to A1's **unified state layer** (atomic appends, locked config writes, single-writer model) — the fix for rank-1 file races.
