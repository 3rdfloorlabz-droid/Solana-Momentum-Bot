## 5. Cadence / Trigger Policy (Manual Only — No Automation)

**No cron, no background loop, no scheduled Node task.** Re-checks are **event-triggered** or **optional calendar reminders** for humans.

### Required triggers (run re-check before proceeding)

| Trigger | Rationale |
|---------|-----------|
| Before any **A4-dependent governance decision** | Decision must reflect current guards, not stale verified state |
| Before any **OR promotion consideration** | A4 verified is context only; guards must still hold if cited |
| After **code changes** to: `runtime_evidence.js`, `runtime_health.js`, `dashboard_server.js`, `a4_approval.js`, `a4_approval_emit.js`, `audit_writer.js`, `a4_rpc_proof.js`, or A4 test suites | Classifier/evidence path may have changed |
| After **dashboard restart** or runtime-health endpoint changes | Stale process / code load risk (A4.35 lesson) |
| After **approval emit, revocation, or expiry change** | Newest approval/revocation event wins |
| After **proof evidence regression** (fallback, failure, stale proof, provider/class change) | Blockers or stability loss may revert posture |
| After **A4 blocker posture change** | NOT_CONFIGURED / REFUSAL / FALLBACK / PROOF_FAILED / PROOF_STALE |

### Optional triggers (recommended hygiene)

| Trigger | Suggested cadence |
|---------|-------------------|
| Project remains active with ongoing A4 governance work | **Manual weekly** spot-check (operator calendar reminder only) |
| Long idle period with no code changes | **Manual monthly** spot-check |
| Before citing A4 status in external status reports | Ad hoc read-only re-check |

### Explicit non-triggers (do **not** start processes for re-check)

- Do **not** start scanner/executor to “refresh” A4 status.
- Do **not** re-run RPC proof unless a separate authorized proof gate says so.
- Do **not** re-emit approval unless a separate human governance gate says so.

## 7. Failure Handling

**Rule: never force verified status.** If any guard fails, the posture is **not currently verified** regardless of prior A4.35 success.

| Failure class | Typical cause | Label | Follow-up gate (examples) |
|---------------|---------------|-------|---------------------------|
| F1 — Approval absent/revoked | Missing event; newer `revoked` row | `A4_NOT_CURRENTLY_VERIFIED` | Human re-approval governance gate + emit |
| F2 — Approval expired | Past `expiresAtIso` | `A4_APPROVAL_EXPIRED` | Re-affirmation / re-emit gate |
| F3 — EvidenceRef mismatch | Provider/class changed | `A4_EVIDENCE_REF_MISMATCH` | Reconcile config + re-emit with matching ref |
| F4 — Stability lost | Proofs stale/failed/insufficient | `A4_STABILITY_REGRESSED` | Authorized read-only proof re-run gate (separate) |
| F5 — Proof scan unavailable | Scan error / audit unreadable | `A4_PROOF_SCAN_UNAVAILABLE` | Audit hygiene / scan failure gate |
| F6 — Blocker active | NOT_CONFIGURED / FALLBACK / etc. | `A4_BLOCKER_ACTIVE` | Blocker-specific remediation gate |
| F7 — SecretRisk | Forbidden field in event | `A4_SECRET_RISK_DETECTED` | Audit sanitization / incident gate |
| F8 — Classifier/code drift | Code changed; tests failing | `A4_CLASSIFIER_REGRESSION` | Fix + regression gate |

**On failure:**

1. Document **exact failing guard** (G1–G10 or F-class).
2. Do **not** cite A4 as verified in governance decisions until re-check passes.
3. Do **not** auto-revert human decision A4.24 — machine posture may lag or regress; governance record stays unless explicitly revoked.
4. Open the **smallest follow-up gate** for the failing class only (one gate at a time).
