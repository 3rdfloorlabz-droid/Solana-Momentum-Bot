## 2. Planning Decision (Encoded)

> **Do not run scanner + executor continuously merely to preserve `HEALTHY_DRY_RUN`.**
> Treat `HEALTHY_DRY_RUN` (branch 4) as an **active-observation posture**.
> Treat `CAPITAL_SAFE_BUT_RUNTIME_AMBIGUOUS` (branch 6) as **acceptable idle-safe posture** when no observation window is active and `capitalExposure` remains `none`.
> Plan a separate B2A/R7b observation run only when sustained paper-trading data collection is actually desired.

The real fork A4.29 resolves is not "how do we keep the dashboard green" but "do we want continuous dry-run data collection right now, or is runtime-health an on-demand verification tool." Decision: **on-demand verification tool at idle; continuous collection only inside deliberate observation windows.**

## 3. Posture 1 — Idle-Safe Posture

The default steady state between observation windows.

| Element | Idle-safe expectation |
|---------|----------------------|
| Dashboard (`dashboard_server.js`) | May remain running |
| Monitor (`monitor.js`) | May remain observational |
| Scanner evidence producer (`scanner_gmgn_trending.js --watch`) | **Not required** to run continuously |
| Executor loop (`live_executor.js --loop`) | **Not required** to run continuously |
| `executor_singleton.lock.json` | May be **absent** — expected at idle |
| `scanner_health.json` | May be stale — expected at idle |
| Classification | Branch 6 `CAPITAL_SAFE_BUT_RUNTIME_AMBIGUOUS` is **acceptable** when no observation run is active |
| Capital exposure | **none** — no exposure enabled |
| `supportsLiveReadiness` | Remains **false** |
| `supportsSoakClaim` | May be **false** — active loop/scanner evidence absent at idle |

**Semantics:** branch 6 at idle is not automatically bad. **This is not a failure state by itself.** It means runtime evidence is stale/missing while capital exposure is none. It is a truthful statement — "nothing is running, nothing is at risk" — not a fault condition to be engineered away.

## 4. Posture 2 — Active Observation Posture

Used only during deliberate paper-trading / observation windows (B2A / R7b-style data collection).

| Element | Active-observation expectation |

| `supportsSoakClaim` | May become **true** as a **machine classifier capability flag only** — **not** human soak authorization |
| Purpose | Deliberate paper-trading data collection — **not dashboard cosmetics** |

**Semantics:** branch 4 is evidence that the dry-run scanner + executor loop is currently healthy. It proves something only while an observation window is open. Outside a window, its absence proves nothing negative.

## 5. A4.28 Mental-Model Correction (Recorded)

A4.28 §9 implied manual lock removal was a required unblock. Corrected:

1. `Stop-Process -Force` bypassing graceful SIGINT shutdown is **normal Windows behavior**, not a defect.
2. The stale-lock TTL (`isExecutorLockStale`, ~3 minutes) already **self-heals** orphan locks; the system was never blocked.
3. Manual removal of `executor_singleton.lock.json` after A4.28 was **cleanup hygiene, not a required unblock**.
4. **Do not build extra lock-cleanup tooling** unless future evidence demonstrates a real problem (e.g., TTL failing to expire, lock contention during legitimate starts).
