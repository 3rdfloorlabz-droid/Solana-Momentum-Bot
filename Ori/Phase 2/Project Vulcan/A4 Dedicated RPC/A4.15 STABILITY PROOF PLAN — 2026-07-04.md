## 3. What Stability Means

Stability is defined as repeated safe evidence exhibiting **all** of:

- repeated proof success (`READ_ONLY_RPC_OK`)
- separated in time (not a burst / not the same call)
- same safe `providerLabel` across successes
- same `endpointClass = dedicated`
- `publicFallbackUsed = false` throughout
- `secretSafe = true` throughout
- no `A4_PROOF_FAILED` (`READ_ONLY_RPC_FAILED`) interleaved between the successes
- no `A4_FALLBACK_DETECTED`
- no stale evidence (all counted proofs within freshness window)
- no scanner/runtime contradiction
- documented receipt for each proof
- explicit human approval before verified status

## 4. Proposed Stability Threshold (first pass)

Minimum A4 stability **candidate** requires:

- at least **2** successful read-only RPC proof events (`READ_ONLY_RPC_OK`)
- at least **15 minutes** apart (time separation, not a burst)
- both within a **24-hour** freshness window (matches current `A4_PROOF_STALE_MS`)
- same `providerLabel`
- `endpointClass = dedicated`
- `publicFallbackUsed = false`
- `secretSafe = true`
- no `A4_PROOF_FAILED` between the two successes
- no `A4_FALLBACK_DETECTED`
- no stale-scanner conflict
- no secret-safety violation
- documented receipt(s)
- explicit human approval (separate step) before `A4_VERIFIED_DEDICATED`

Notes / non-forcing:
- These are conservative first values. If future evidence suggests a stronger
  bar (e.g. 3 successes, or a 30-minute separation, or a rolling 48h window),
  adjust in A4.16+ rather than treating these as final.
- The 15-minute separation intentionally rejects rapid re-runs; it is a
  "different moment in time" signal, not a performance metric.

## 6. Evidence Model (future runtime_evidence plan)

Planned additive logic (not implemented here):

- collect **all** `A4_READ_ONLY_RPC_PROOF` events from the audit tail (not just
  newest), reusing the existing per-row `extractA4ProofFacts` allowlist/reject
  guards
- sort by timestamp ascending
- allowlist-sanitize each; reject stale or malformed events
- reject any event carrying forbidden/secret-shaped fields (already enforced)
- require same `providerLabel` and `endpointClass` across counted successes
- require no `READ_ONLY_RPC_FAILED` and no fallback event interleaved between
  the counted successes
- produce a **stability summary only** (never raw values)

Allowed future stability summary fields:
- `successCount`
- `firstProofAt`
- `latestProofAt`
- `separationBucket` (e.g. `<15m`, `15-60m`, `1-24h`, `>24h`)

- `endpointClassConsistent` (bool)
- `fallbackObserved` (bool)
- `failureObserved` (bool)
- `secretSafe` (bool)

Forbidden anywhere in the stability summary:
- raw endpoint / url / uri

- headers
- wallet
- signature
- transaction
- `process.env`

Design note: keep the existing single-newest `proof` field intact and add a
separate `proofStability` object, so the A4.13 behavior is not weakened.

## 7. Runtime Health Boundary (future runtime_health plan)

- **Threshold not met** → remain `A4_READ_ONLY_RPC_VERIFIED`;
  `recommendedAction`: run a time-separated stability proof under human approval.
- **Threshold met, not approved** → `A4_STABILITY_PROOF_OBSERVED`;
  `blockerActive = true`; `supportsLiveReadiness = false`;
  `supportsSoakClaim = false`.
- **Human approval recorded later** → status may become `A4_VERIFIED_DEDICATED`;
  still `supportsLiveReadiness = false` unless separate gates (A1 and other
  blockers) clear independently. A4 alone never grants readiness.

Never auto-emit `A4_VERIFIED_DEDICATED` from evidence alone.

## 8. Verification Cadence Options

### Option A — Manual second proof only
- Pros: safest; explicit human authorization; no automation; low surface area.
- Cons: slower; operator must remember.

### Option B — Scheduled manual prompt / reminder
- Pros: preserves human gate; improves consistency.
- Cons: adds process overhead.

### Option C — Automatic recurring proof
- Pros: best operational visibility.
- Cons: creates active probing; rate/secret/logging surface; premature
  automation; conflicts with the "no recurring proof" boundary held so far.

**Recommendation: Option A first.** No automatic recurring proof. Option B may
be layered later as a non-executing reminder only.
