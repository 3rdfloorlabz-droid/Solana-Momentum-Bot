## 3. What Stability Evidence Means

Stability evidence means **repeated infrastructure proof exists**.

It does **NOT** mean:

- A4 resolved
- live readiness
- strategy validity
- scanner edge proven
- signer safety
- capital readiness
- A1 resolved

## 4. Current Implementation Gap (read-only findings)

Observed in `runtime_evidence.js` (no changes made):

- `extractA4ProofFacts(row)` ‚Äî already sanitizes a single proof row, rejects
  wrong producer/eventType, forbidden keys, and secret-shaped strings. **Reusable
  as-is** for multi-event evaluation.
- `buildA4ProofEvidence(auditRows, options)` ‚Äî collects all valid proof facts but
  then **reduces to the single newest** (`facts.reduce(... latest)`) and returns
  one `proof` object with `freshness`.
- `buildA4Evidence(...)` ‚Äî elevates posture only from that single newest proof
  (fresh + OK + secretSafe + not-fallback ‚Üí `A4_READ_ONLY_RPC_VERIFIED`).

Gap: there is no aggregation across multiple proofs (count, separation,
consistency), so two successes look identical to one. A4.17 plans an **additive**
stability summary that does not weaken the existing single-proof `proof` field.

## 5. Proposed Evaluation Model (future `runtime_evidence`)

Add an additive `buildA4ProofStability(auditRows, options)` (name indicative)
that:

1. reuses `extractA4ProofFacts` for every proof row (same allowlist/reject rules)
2. drops any row flagged `secretRisk` (and records `secretExposureRisk`)
3. keeps only rows within the freshness window (reuse `A4_PROOF_STALE_MS = 24h`)
4. sorts kept facts by timestamp ascending
5. counts consecutive successes and inspects interleaving
6. computes consistency (same `providerLabel`, same `endpointClass`)
7. computes time separation between the two most recent successes
8. produces a **stability summary only** ‚Äî never raw values

Design constraints:
- purely additive; leave the existing `proof` (newest) field intact
- expose a new sibling object `proofStability` on `a4Evidence`
- never elevate beyond `A4_STABILITY_PROOF_OBSERVED` (no auto verified-dedicated)
- never touch config/refusal/fallback detection precedence

### Allowed stability summary fields

- `successCount`
- `firstProofAt`
- `latestProofAt`
- `separationBucket` (`<15m`, `15-60m`, `1-24h`, `>24h`, `unknown`)
- `providerConsistent` (bool)
- `endpointClassConsistent` (bool)
- `providerLabel` (allowlisted label only, when consistent; else `mixed`/`unknown`)
- `endpointClass` (when consistent; else `mixed`/`unknown`)
- `fallbackObserved` (bool)
- `failureObserved` (bool)
- `secretSafe` (bool ‚Äî all counted rows safe)
- `withinFreshnessWindow` (bool)
- `stabilityCandidate` (bool ‚Äî threshold met, NOT approval)

### Forbidden anywhere in the summary

raw endpoint / url / uri, raw slot value, API key, headers, wallet, signature,
transaction, `process.env`, stack trace.

## 6. Proposed Threshold (carried from A4.15, refined)

`stabilityCandidate = true` requires **all**:

- `successCount >= 2`
- most-recent-two successes `>= 15 minutes` apart (not a burst)
- both within the 24h freshness window
- `providerConsistent === true`
- `endpointClassConsistent === true` and class is `dedicated`
- `fallbackObserved === false`
- `failureObserved === false` (no `READ_ONLY_RPC_FAILED` interleaved between the
  counted successes)
- `secretSafe === true` for every counted row

These match the A4.16 evidence already on disk. Values remain first-pass; a
future gate may raise to 3 successes or widen separation without breaking this
model.

## 7. Proposed Runtime Health Mapping (future `runtime_health`)

Additive status handling; all keep readiness invariants false.

| Condition | Status | severity | blockerActive |
|-----------|--------|----------|---------------|
| one fresh success, threshold not met | `A4_READ_ONLY_RPC_VERIFIED` (unchanged) | warning | true |
| `stabilityCandidate === true`, no human approval | `A4_STABILITY_PROOF_OBSERVED` | warning | true |
| threshold met **and** recorded human approval | `A4_VERIFIED_DEDICATED` (future gate only) | info | true (infra-only) |
| any interleaved failure | `A4_PROOF_FAILED` (existing) | blocked | true |
| newest proof stale | `A4_PROOF_STALE` (existing) | warning | true |

Rules:
- `A4_STABILITY_PROOF_OBSERVED` may **only** be produced when the current
  config-derived status is `A4_READ_ONLY_RPC_VERIFIED` / `A4_CONFIGURED_UNVERIFIED`
  (never from `NOT_CONFIGURED` / `REFUSAL_ACTIVE` / `FALLBACK_DETECTED`).
- `A4_VERIFIED_DEDICATED` is **never** auto-emitted from evidence alone; it
  requires a recorded human-approval signal (see ¬ß10) that runtime_health reads
  as an explicit, separate input ‚Äî not inferred from proof counts.
- `supportsLiveReadiness = false` and `supportsSoakClaim = false` for every A4
  status, unchanged.
- `recommendedAction`:
  - not met ‚Üí "Run a time-separated read-only stability proof under approval."
  - observed ‚Üí "Stability threshold met; human approval required before
    A4_VERIFIED_DEDICATED. Not live readiness."

Forbidden display fields: RPC URL, API key, headers, raw slot, wallet, signature,

Sanitize through a positive allowlist (mirror `sanitizeA4ProofForDisplay`).

## 10. Human Approval Model (carried from A4.15)

- Require an explicit **decision doc** before `A4_VERIFIED_DEDICATED`, e.g.
  `Ori/Decisions/DECISION ‚Äî <date> ‚Äî A4 Stability Proof Accepted.md`.
- Runtime health should treat approval as an **explicit input signal** (e.g. a
  dedicated, attributed approval record it can read), never inferred from proof
  counts alone.
- Do not create the decision doc now. Approval is a deliberate human step.

Open design question for A4.18: how the approval signal is represented safely
(decision doc reference vs. a specific attributed audit event) so runtime_health
can read it without granting itself authority.

## 11. A4 Resolution Boundary

Even when `stabilityCandidate` is true and human approval is recorded,
`A4_VERIFIED_DEDICATED` remains an **infrastructure** status only.

It does not mean: A1 resolved, live readiness, execution safety, strategy
validity, capital readiness, signer safety, or scanner edge proven. A4 can
support future live-readiness reviews but cannot grant them alone.
