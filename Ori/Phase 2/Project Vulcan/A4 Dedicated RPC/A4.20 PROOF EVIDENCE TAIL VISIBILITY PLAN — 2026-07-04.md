## 3. Root Cause

- The issue is **not** proof validity — both proofs are `READ_ONLY_RPC_OK`, dedicated, secret-safe, non-fallback.
- The issue is **not** stability evaluator logic — A4.18 maps correctly when rows are reachable.
- The issue is **not** RPC connectivity — no network dependency here.
- The issue **is** evidence discovery.

A4 proof events are **rare governance/audit events**. Monitor / `live_executor` events are **frequent**. A small, generic, count-based audit tail hides rare-but-important evidence behind common high-frequency rows. Evidence that exists but is unreachable is not runtime truth.

## 4. Design Goals

A future visibility fix should:

- keep proof evidence discoverable regardless of monitor volume
- remain **bounded** (predictable, capped work)
- avoid parsing the entire log on every health request
- avoid active probing / no RPC / no proof re-run
- avoid any new secret exposure surface
- avoid overclaiming A4 (no auto-promotion, no verified-dedicated)
- preserve normal runtime performance
- remain fully testable with fixtures
- preserve `supportsLiveReadiness: false` and `supportsSoakClaim: false` invariants

## 5. Design Options

### Option A — Increase global `auditTailLimit`

Raise the default from 50 to e.g. 500 or 5000 for all evidence collection.

Pros:
- trivial change
- likely surfaces proof rows short-term

Cons:
- broad blast radius (affects scanner/lock/heartbeat/producer summaries too)
- performance cost grows with log size on every request
- not semantically targeted — monitor volume can bury proof rows again as the log grows
- makes all evidence collection heavier for one rare signal

Recommendation: **Not preferred** as the primary fix.

### Option B — Targeted bounded proof-event scan (preferred)

Keep the general audit tail small (50). Add a **separate, bounded, targeted** scan that reads a larger recent window and filters only A4 proof events.

Behavior:
- general audit tail remains 50 (unchanged for all existing consumers)
- add a targeted scan reading the last N lines (proposed 5000)
- filter only `producer === "a4_rpc_proof"` and `eventType === "A4_READ_ONLY_RPC_PROOF"`
- reuse existing `extractA4ProofFacts` sanitizer / allowlist
- reuse existing `buildA4ProofEvidence` (newest) and `buildA4ProofStability` (multi)
- return allowlisted metadata only; ignore all other rows without exposing their content

Pros:
- semantically targeted at the rare signal
- bounded and predictable
- preserves all existing general-evidence behavior
- robust against monitor noise
- no new state file, no cache reconciliation
- no proof re-run, no RPC
- stays audit-source-of-truth aligned

Cons:
- reads more lines for the proof scan specifically
- needs tests and an explicit performance boundary

Recommendation: **Preferred.**

### Option C — Dedicated A4 proof status/cache file

Write latest proof/stability state to a small JSON file that runtime_evidence reads.

Pros:
- very fast read
- immune to audit-tail burial

Cons:
- introduces a new state lifecycle to maintain and trust
- stale-cache risk and cache/audit reconciliation problem
- another file that can disagree with the audit source of truth
- weakens the audit-backed evidence model

Recommendation: **Later only** if the targeted scan proves insufficient.

### Option D — Re-run proof when visibility is missing

Trigger a fresh proof when runtime health can't see one.

Pros:
- proof becomes fresh

Cons:
- active RPC call inside a read-only health path
- masks the evidence-window problem instead of fixing it
- creates recurring proof pressure / rate + surface-area growth
- inappropriate for `runtime_health` (must stay side-effect free)

Recommendation: **Reject** for this gate (and as a health-path behavior generally).

## 6. Recommended Design

Adopt **Option B — targeted bounded A4 proof-event scan** in `runtime_evidence`.

Suggested shape (future implementation, not authorized here):

- keep existing `auditTailLimit` (default 50) for general evidence untouched
- add a separate option, e.g. `a4ProofScanLimit` (a.k.a. `a4ProofAuditScanLimit`)
  - default proposed: **5000 lines**
- the targeted scan:
  - reads only the last `a4ProofScanLimit` lines of `execution_audit.jsonl`
  - keeps only rows with `producer === "a4_rpc_proof"` and `eventType === "A4_READ_ONLY_RPC_PROOF"`
  - runs them through the existing `extractA4ProofFacts` allowlist/sanitizer
  - feeds the existing `buildA4ProofEvidence` (newest) and `buildA4ProofStability` (multi) builders
  - returns allowlisted metadata only; never prints forbidden fields; never calls RPC; never runs proof

Important: **Do not make this `.env`-configurable** in the first pass unless strongly justified. Keep it a code default with an injectable option for tests, matching the A4.18 threshold-constant approach.

---

## 7. Boundaries and Performance

Guardrails for the targeted scan:

- **bounded line count** — never unbounded; cap at `a4ProofScanLimit`
- **no full-file parse** on the normal health path; a full-file diagnostic mode may exist but must be explicitly requested (as in A4.19's read-only diagnostic)
- **fast rejection** of non-A4 rows (cheap producer/eventType check before deeper parsing)
- **never expose ignored row content** — dropped rows contribute nothing to output
- **tests use fixtures only** — synthetic audit files, no real audit, no RPC, no `.env`
- **fail-safe on error / oversize:**
  - proof → `null`
  - proofStability.stabilityCandidate → `false`
  - a safe `recommendedAction` such as "proof evidence unavailable (scan window/read error)"
  - never throw into the health response; degrade to conservative posture

Scan-window sizing rationale:
- A4.19 measured the newest proof row at **114 lines** from the tail with monitor volume active.
- **5000 lines** provides ~44× current headroom and covers substantial monitor bursts.
- If monitor cadence is high enough that 5000 lines spans less than the 24h freshness window, prefer **10000** as a conservative default. Recommendation: start at **5000**, and document that A4.21 verification should measure the observed line-distance of proof rows and bump to 10000 if headroom is thin. Do not exceed a bounded cap that risks per-request latency on a large append-only log.
