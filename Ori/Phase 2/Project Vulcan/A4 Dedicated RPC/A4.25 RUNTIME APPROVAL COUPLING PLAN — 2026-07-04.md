## 3. Approval Source of Truth Options

### Option A — Parse the decision markdown directly

Runtime reads `Ori/Decisions/DECISION — 2026-07-04 — A4 Stability Proof Accepted.md` and interprets the approval block.

Pros:
- human-readable, single artifact
- direct governance trace
- no duplicate approval state

Cons:
- markdown parsing is brittle (checkbox formats, wording drift)
- workspace path uncertainty (canonical OneDrive vs workspace-local Ori; runtime root differs from the Ori tree)
- harder to test deterministically
- filename contains an em dash + date; fragile to reference from code

Recommendation: **Not preferred** as the machine signal.

### Option B — Explicit machine-readable approval marker (attributed audit event) — preferred

A human-run approval tool (separate future gate) emits ONE attributed, secret-free approval event into the existing audit stream, mirroring the proof-event model:

- `producer: "a4_approval"`
- `eventType: "A4_VERIFIED_DEDICATED_APPROVAL"`
- allowlisted payload only: `approver` (label, not secret), `approvalStatus` ("approved" / "approved_with_conditions" / "revoked"), `decisionRef` (stable slug/id of the decision doc, not a full path), `evidenceRef` (safe snapshot id), `approvedAtIso`, optional `expiresAtIso`

Pros:
- reuses the existing audit-source-of-truth + targeted-scan machinery (A4.21)
- same sanitizer/allowlist discipline as proof events (no secret surface)
- deterministic, fixture-testable
- naturally supports revocation (a later `revoked` event supersedes)
- decouples human narrative (markdown) from machine signal (audit event)

Cons:
- requires a small human-run emit tool (separate gate)
- the markdown doc and the audit event must be kept consistent (the doc is the human record; the event is its machine echo)

Recommendation: **Preferred.**

### Option C — Small dedicated approval marker JSON file

A human-authored `a4_approval.json` (or under `Ori/Decisions/`) with allowlisted fields, read by runtime.

Pros:
- simple to read and test
- no markdown parsing

Cons:
- a second state file to trust and reconcile with the decision doc
- no built-in append-only history (revocation overwrites rather than supersedes)
- easier to edit silently than an attributed audit event
- weaker provenance than an attributed event

Recommendation: **Acceptable fallback** if an audit-event emit tool is undesirable; Option B is stronger for provenance and revocation history.

## 4. Recommended Design

Adopt **Option B — explicit machine-readable approval marker as an attributed, secret-free audit event**, discovered via a targeted bounded scan analogous to A4.21's proof scan.

Illustrative (future implementation, NOT authorized here):

- A separate human-run tool (its own approval gate) appends one `A4_VERIFIED_DEDICATED_APPROVAL` event when Taylor signs the decision doc.
- `runtime_evidence` gains a bounded `readA4ApprovalScan()` (mirroring `readA4ProofScan`) that keeps only `producer === "a4_approval"` / `eventType === "A4_VERIFIED_DEDICATED_APPROVAL"` rows, sanitizes via a positive allowlist, and returns the newest valid approval/revocation state.
- `a4Evidence` gains an additive, secret-safe `approval` summary: `{ present, status, approver, decisionRef, evidenceRef, approvedAtIso, expiresAtIso, freshness }` — allowlisted only.
- The markdown decision doc remains the **human** record; the audit event is its **machine echo**. The doc is never parsed by runtime.

Guardrails:
- runtime **never** self-authors an approval event (no auto-write anywhere)
- absence/ambiguity/expiry/revocation → fail-closed (stay `A4_STABILITY_PROOF_OBSERVED`)
- approval alone still does not grant live/soak readiness

## 5. Approval Recognition Preconditions

`runtime_health` may map to `A4_VERIFIED_DEDICATED` **only when ALL hold**:

- base posture would otherwise be `A4_STABILITY_PROOF_OBSERVED`
- `proofStability.stabilityCandidate === true`
- `proofScan.available === true`
- a valid approval event is present with `status: "approved"` (or `"approved_with_conditions"`)
- the approval references current-class evidence (`evidenceRef` consistent with the visible provider/endpoint class)
- approval is not expired (if `expiresAtIso` present) and not superseded by a later `revoked` event
- no active blocker (`NOT_CONFIGURED` / `REFUSAL_ACTIVE` / `FALLBACK_DETECTED` / `PROOF_FAILED` / `PROOF_STALE`)

If any precondition fails → remain `A4_STABILITY_PROOF_OBSERVED` (or the appropriate blocked posture).

---

## 6. Fail-Closed Conditions

Runtime must NOT emit `A4_VERIFIED_DEDICATED` if:

- no approval event is present
- approval status is `PENDING_REVIEW` / `not_approved` / `revoked`
- approval is expired or stale beyond the accepted window
- the approval event carries any forbidden/secret-shaped field (reject + flag, no exposure)
- stability evidence regresses (fallback/failure/stale/not-configured)
- the approval cannot be parsed or the scan fails (degrade safely)

Default on any doubt: `A4_STABILITY_PROOF_OBSERVED`.

---

## 7. Revocation / Expiry Model

- **Supersession:** the newest attributed approval/revocation event wins; a later `revoked` event immediately returns runtime to `A4_STABILITY_PROOF_OBSERVED`.
- **Expiry:** optional `expiresAtIso`; past expiry → treated as not approved (re-affirmation required).
- **Evidence regression:** a later `PROOF_FAILED` / `PROOF_STALE` / fallback re-blocks regardless of a prior approval.
- **Config/provider change:** a change in provider label or endpoint class from the approved class invalidates recognition (evidenceRef mismatch).

Verified is a **current** statement, never permanent.
