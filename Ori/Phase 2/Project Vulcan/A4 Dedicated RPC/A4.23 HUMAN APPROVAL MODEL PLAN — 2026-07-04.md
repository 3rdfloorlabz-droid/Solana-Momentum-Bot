## 3. Why Evidence Is Not Approval

- Evidence answers: *"Is the dedicated RPC path demonstrably working, repeatedly, without fallback or secret exposure?"* — **Yes** (A4.18–A4.22).
- Approval answers: *"Does an accountable human accept this evidence as sufficient to mark A4 verified, knowing what that status does and does not authorize?"* — **Not yet.**

Auto-promoting on evidence alone would:
- collapse the distinction between *observed* and *accepted*
- remove the human accountability checkpoint before an infra-trust status
- risk status creep (verified → assumed live-ready) if the boundary is not human-anchored

Principle: **Stable evidence is not approval. Approval must be explicit.**

## 5. Approval Model Options

### Option A — Signed approval record file (audit-backed decision doc)

A human writes/commits an approval decision doc (e.g. `Ori/Decisions/…`) capturing approver identity, timestamp, evidence snapshot reference, and explicit scope. Runtime reads a safe pointer to confirm approval exists.

Pros:
- explicit, durable, reviewable
- audit-source-of-truth aligned
- decouples "evidence visible" (runtime) from "evidence accepted" (human)
- no secret surface

Cons:
- needs a defined schema and a safe runtime read path
- must guard against forged/auto-generated approval

Recommendation: **Preferred.**

### Option B — Dashboard approval toggle / POST route

An operator clicks an "approve A4" control that flips a runtime flag.

Pros:
- fast, in-context

Cons:
- POST control-route surface (explicitly out of scope for these gates)
- weaker durable audit trail / accountability
- easy to click without a recorded rationale
- conflicts with the "no dashboard control endpoints" posture

Recommendation: **Reject** for first pass.

### Option C — Config-file flag (e.g. in live_config.json)

Set an `a4Approved: true` field.

Pros:
- simple to read

Cons:
- co-locates governance approval with operational config (mixing concerns)
- no approver identity / rationale / evidence linkage
- easy to set silently; weak accountability

Recommendation: **Reject** (governance ≠ operational config).

### Option D — Manual status assertion only (no runtime coupling)

Approval lives only in docs; runtime never emits `A4_VERIFIED_DEDICATED`.

Pros:
- zero code risk

Cons:
- runtime and governance stay permanently disconnected
- dashboard can never reflect the accepted state

Recommendation: **Acceptable interim**, but Option A is the target.

## 6. Recommended Approval Model

Adopt **Option A — signed approval record (audit-backed decision doc)** with a safe runtime read path (to be implemented under a **separate future gate**, not here).

Shape (future, illustrative — not authorized here):

- an approval decision doc under `Ori/Decisions/` (canonical) or workspace-local `Ori/Decisions/` capturing:
  - approver identity (human name/handle)
  - approval timestamp (ISO)
  - initiative reference (`A4.x`)
  - **evidence snapshot reference** (the A4.22-style safe summary: successCount, separationBucket, providerLabel, endpointClass, proofScan.available/limit — no secrets)
  - explicit scope statement (what verified does and does not authorize)
  - revocation note field
- a small, safe **runtime approval pointer** the code can read (e.g. an attributed audit event `A4_VERIFIED_DEDICATED_APPROVAL` written by a human-run tool, or a minimal approval marker file) — carrying only allowlisted, secret-free fields.

Guardrails:
- runtime must **never** self-author the approval (no auto-write)
- approval must reference the evidence it accepts (traceability)
- absence/ambiguity of approval → remain `A4_STABILITY_PROOF_OBSERVED` (fail-closed)
- approval alone still does not grant live/soak readiness

Before a human approval is valid, runtime evidence should show:

- `a4Health.status === A4_STABILITY_PROOF_OBSERVED`
- `proofStability.stabilityCandidate === true`
- `successCount >= 2`, `freshSuccessCount >= 2`
- `separationBucket === ">=15m"`
- `providerConsistent === true`, `endpointClassConsistent === true`
- `endpointClass === "dedicated"`
- `fallbackObserved === false`, `failureObserved === false`
- `secretSafe === true`
- `proofScan.available === true`
- no active blocker (`NOT_CONFIGURED` / `REFUSAL_ACTIVE` / `FALLBACK_DETECTED` / `PROOF_FAILED` / `PROOF_STALE`)

If any precondition fails after approval (e.g. later fallback/failure), the approval must not keep asserting verified — see revocation.

## 9. Revocation / Expiry Model

Approval is not permanent trust:

- **Evidence regression:** if a later `PROOF_FAILED`, `PROOF_STALE`, or fallback appears, runtime must drop back to the appropriate blocked/observed posture regardless of a prior approval marker.
- **Staleness:** consider an approval freshness window (e.g. re-affirm if the accepted evidence ages beyond a defined bound) — to be finalized in the implementation gate.
- **Explicit revocation:** a human may record a revocation marker; runtime treats it as fail-closed immediately.

Verified is a **current** statement, not a permanent one.

---

## 10. Secret Safety Boundary

The approval model must never introduce a secret surface:

- approval records carry **no** RPC URL, API key, headers, raw slot, wallet, signature, transaction, or `process.env`
- evidence snapshot in the approval doc uses the same allowlisted safe fields already surfaced by A4.18/A4.21 (labels, buckets, booleans, counts)
- runtime approval pointer is allowlisted and sanitized like existing proof/stability summaries
