## 2. What `A4_VERIFIED_DEDICATED` Now Means

`A4_VERIFIED_DEDICATED` is an **additive, infrastructure-only, secret-safe runtime/governance status** on the A4 dedicated-RPC evidence path. It means all of the following are true **at classification time**:

1. **A4.24 was approved by Taylor** (A4.34 sign-off captured; decision record `APPROVED_BY_TAYLOR`; scope = stability-evidence-only).
2. **A4.25 approval coupling was implemented and tested** — bounded approval scan, `buildA4ApprovalEvidence`, guarded `classifyA4Health()` elevation; runtime never self-authors approval.
3. **A4.35 verified the approval-coupling path in live `/api/runtime-health`** — human-run `A4_VERIFIED_DEDICATED_APPROVAL` audit event emitted; live `a4Health.status` confirmed `A4_VERIFIED_DEDICATED` after dashboard restart.
4. **The live runtime/governance evidence path can now represent the approved stability proof**, subject to ongoing guard checks:
   - repeated safe read-only RPC proof stability evidence (`proofStability.stabilityCandidate === true`)
   - targeted proof scan available (`proofScan.available === true`)
   - explicit attributed approval present and approved (`approval.present && approval.approved`)
   - approval evidenceRef matches current provider/endpoint class (`approval.evidenceRefConsistent === true`)
   - approval not expired; no secretRisk; no active A4 blocker posture

**Plain-language statement:**

> Taylor explicitly approved the A4.24 stability-evidence decision; that approval is now machine-visible in the audit stream; the runtime classifier recognizes it and reports that dedicated-RPC **stability evidence** is verified — **not** that the bot is cleared for live trading, human soak, capital exposure, or OR promotion.

**Verified is current, not permanent.** Revocation, expiry, evidence regression (fallback/failure/stale), provider/class change, or blocker reactivation can return posture to `A4_STABILITY_PROOF_OBSERVED` or a blocked state per A4.25 fail-closed rules.
