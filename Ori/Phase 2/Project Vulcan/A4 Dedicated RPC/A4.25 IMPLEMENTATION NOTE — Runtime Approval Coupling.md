## Summary

Minimal safe coupling between the approved A4.24 human decision and the runtime/governance evidence path was implemented per A4.25 Option B: an attributed, secret-free `A4_VERIFIED_DEDICATED_APPROVAL` audit event is discovered via a bounded scan; `runtime_health` may map `A4_STABILITY_PROOF_OBSERVED` → `A4_VERIFIED_DEDICATED` only when all A4.25 preconditions hold. Runtime never self-authors approval. Live/soak readiness and capital exposure invariants preserved.

## Coupling Behavior

1. **`a4_approval.js`** — builds secret-safe approval audit events (`producer: a4_approval`, `eventType: A4_VERIFIED_DEDICATED_APPROVAL`). Does not write files or emit runtime status by itself.

2. **`runtime_evidence.js`** — `readA4ApprovalScan()` (default limit 5000, mirroring proof scan), `mergeA4ApprovalRows()`, `buildA4ApprovalEvidence()`; `collectRuntimeEvidence()` / `buildA4Evidence()` carry additive `approval` + `approvalScan` summaries on `a4Evidence`.

3. **`runtime_health.js`** — guarded elevation in `classifyA4Health()`:
   - Requires base posture `A4_STABILITY_PROOF_OBSERVED` (via stability candidate from `A4_READ_ONLY_RPC_VERIFIED`)
   - Requires `proofStability.stabilityCandidate === true`
   - Requires `proofScan.available === true`
   - Requires `approval.present && approval.approved && approval.evidenceRefConsistent`
   - Requires approval not expired and no `secretRisk`
   - Blockers (`NOT_CONFIGURED`, `REFUSAL_ACTIVE`, `FALLBACK_DETECTED`, `PROOF_FAILED`, `PROOF_STALE`) always win
   - **`supportsLiveReadiness: false`** and **`supportsSoakClaim: false`** even when verified

4. **`dashboard_server.js`** — display-only `approval` + `approvalScan` via positive allowlists; updated verified-dedicated operator messaging (not live readiness).

5. **`a4_approval_emit.js`** — human-run CLI to append one approval audit row (not auto-run in this gate).
